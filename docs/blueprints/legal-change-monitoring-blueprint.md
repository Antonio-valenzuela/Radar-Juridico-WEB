# Radar Jurídico Legal Change Monitoring — Blueprint

> Generated by The Architect on 2026-07-05  
> Archetype: Internal legaltech monitoring dashboard

---

## 1. Project Overview

### Vision

Convertir Radar Jurídico de un repositorio que intenta descargar, extraer e indexar PDFs completos en el flujo web, a un monitor estable de cambios legales basado en metadatos oficiales, `ETag`, `Last-Modified`, tamaño de archivo, hashes, versiones ligeras y alertas revisables por abogados.

La demo debe permitir que un abogado vea documentos clave, confirme cuándo fueron revisados, identifique si hubo cambios, abra la fuente oficial y pregunte a la IA sin que la plataforma invente reformas ni exponga detalles técnicos como Redis, BullMQ, workers, JSON, embeddings o errores internos.

### Goals

- Evitar que el Web Service de Render descargue o extraiga PDFs pesados dentro de requests HTTP.
- Monitorear documentos legales curados usando metadatos ligeros y hashes.
- Reutilizar modelos existentes antes de crear tablas nuevas.
- Mostrar `/monitoreo` y adaptar `/watchlists` como experiencia visual de “Alertas”.
- Hacer que la IA responda primero con cambios detectados y, si no hay eventos, diga claramente que no encontró cambios indexados.

### Success Metrics

- `npm run build`, `npm test`, `npx tsc --noEmit` pasan.
- `npx tsx scripts/monitor-legal-documents.ts --dry-run` muestra documentos revisados, sin cambios, cambios detectados y errores por fuente sin escribir en DB.
- La web sigue cargando dashboard, búsqueda, documentos, Centro Jurídico, IA y alertas sin depender de workers.
- PostgreSQL no guarda PDFs completos ni texto completo del corpus legal.
- Una consulta como “cambios en materia penal de esta semana” no inventa reformas.

## 3. High-Level Architecture

```text
Documentos curados en DB
        |
        v
scripts/monitor-legal-documents.ts
        |
        | HEAD/GET ligero a fuente oficial
        | ETag / Last-Modified / Content-Length / SHA-256
        v
Document + DocumentVersion + DocumentChange + Notification
        |
        +--> /monitoreo muestra estado de documentos
        +--> /watchlists se presenta como Alertas
        +--> IA consulta cambios antes de responder
```

El Web Service nunca ejecuta monitoreo pesado. Solo lee datos ya persistidos, muestra UI, ejecuta búsquedas e IA. El monitor corre manualmente con `npx tsx scripts/monitor-legal-documents.ts` y soporta `--dry-run` para validar sin escribir.

## 4. Existing Models To Reuse

### Reusar sin duplicar

| Modelo | Uso en arquitectura nueva |
|---|---|
| `OfficialSource` | Fuente oficial: Cámara de Diputados, DOF/SIDOF, SCJN/SJF, etc. |
| `Document` | Documento monitoreado canónico: Constitución, Código Penal Federal, CNPCF, etc. |
| `DocumentVersion` | Versión ligera cuando se detecta cambio por hash/metadatos. |
| `DocumentChange` | Evento de cambio asociado a una nueva versión. Usarlo como equivalente de `ChangeEvent`. |
| `DocumentMetadata` | Metadatos extensibles si un campo no merece columna propia. |
| `Watchlist` | Reglas de interés por tema, keyword o norma. Mantener, pero llamarlo “Alertas” en UI. |
| `Notification` | Alertas internas por versión/cambio para demo. |
| `AlertRule` | Reglas avanzadas si se requiere compatibilidad futura con notificaciones por usuario/org. |
| `IngestRun` | Registro de ejecución del monitor por batch. |
| `ProcessingJob` | Opcional para fase futura; no necesario para modo manual inicial. |
| `Item` | Mantener para publicaciones/noticias/registros existentes; no usar como fuente primaria del monitor de leyes curadas. |
| `DocumentChunk` / `Embedding` | No usar en monitoreo ligero; solo cuando exista flujo explícito de RAG/indexación. |

### No crear inicialmente

- No crear `ChangeEvent` si `DocumentChange` cubre el evento con campos mínimos adicionales.
- No crear otra tabla de alertas si `Notification`/`Watchlist` puede cubrir la demo.
- No crear tabla para PDFs ni blobs.

## 5. Data Model Changes

### Prisma Migration: Non-Destructive Only

Crear una migración nueva con `npx prisma migrate dev --name legal_change_monitoring_metadata` en desarrollo local. Revisar el SQL antes de aplicar en producción.

Reglas:

- No `prisma migrate reset`.
- No `DROP TABLE`.
- No `DROP COLUMN`.
- No `TRUNCATE`.
- No `DELETE`.
- Nuevos campos `NOT NULL` solo con `DEFAULT`, o agregarlos primero nullable.
- No guardar PDF completo.
- No guardar texto completo del corpus legal.

### `Document` fields to add

```prisma
model Document {
  // existing fields stay untouched

  shortCode         String?
  matter            String?
  officialSourceId  String?
  officialUrl       String?
  officialSourceUrl String?
  currentHash       String?
  etag              String?
  lastModified      DateTime?
  fileSize          BigInt?
  lastCheckedAt     DateTime?
  lastError         String?
  monitoringStatus  String?
  changeSummary     String?

  officialSource OfficialSource? @relation(fields: [officialSourceId], references: [id], onDelete: SetNull)

  @@index([matter, monitoringStatus])
  @@index([officialSourceId])
  @@index([lastCheckedAt])
  @@index([shortCode])
}
```

Notes:

- `status` already exists on `Document` and may be used for document lifecycle/indexing. Do not repurpose it for monitoring.
- Use `monitoringStatus` for legal monitoring state with values: `active`, `unchanged`, `changed`, `error`, `blocked`.
- `canonicalUrl` already exists; keep it. `officialUrl` is the exact monitored URL. If desired, set both to the same value for curated docs.
- `latestVersionHash` already exists; keep it as compatibility alias. New monitor should update both `currentHash` and `latestVersionHash` unless the team decides to deprecate one later.
- `jurisdiction` already exists; use `"Federal"` or `"MX-Federal"` consistently for UI. Existing default is `"MX"`.

### `DocumentVersion` fields to add or adapt

```prisma
model DocumentVersion {
  // existing fields stay untouched

  etag          String?
  lastModified DateTime?
  fileSize     BigInt?
  sourceUrl    String?
  checkRunId   String?
  metadata     Json?

  @@index([documentId, createdAt])
}
```

Notes:

- `contentHash` already exists and should store the current content/hash signal.
- `rawText` and `originalText` must remain nullable and must not be populated by the lightweight monitor.
- `rawRef` may store a small reference string such as official URL or storage pointer, not a blob.

### `DocumentChange` fields to add or adapt

```prisma
model DocumentChange {
  // existing fields stay untouched

  sourceUrl     String?
  detectedAt    DateTime @default(now())
  previousHash  String?
  newHash       String?
  priority      String   @default("media") // baja | media | alta
  reviewStatus  String   @default("nueva") // nueva | revisada | descartada
  matter        String?
  jurisdiction  String?
}
```

Notes:

- `before` and `after` exist but should stay `null` in monitoring ligero.
- `changeDescription` should be short and lawyer-facing, e.g. `Cambio detectado por ETag/Last-Modified en Código Penal Federal`.
- `changeType`: use `unchanged`, `metadata_changed`, `hash_changed`, `new_document`, `error`, `blocked` where applicable. For `unchanged`, do not create `DocumentChange`; only update `Document`.

### `Notification` for alert rows

Reuse `Notification` for in-app alert rows:

- `organizationId`: demo org.
- `documentVersionId`: new version when change detected.
- `channel`: `"in_app"`.
- `status`: `"nueva" | "revisada" | "descartada"`.
- `payload`: JSON with `documentId`, `documentTitle`, `matter`, `changeType`, `priority`, `changeId`, `officialUrl`.

If `Notification.status` is already used elsewhere, keep status values compatible and handle mapping in UI.

## 7. Monitoring Script

### File

Create:

```text
scripts/monitor-legal-documents.ts
```

### CLI

```bash
npx tsx scripts/monitor-legal-documents.ts
npx tsx scripts/monitor-legal-documents.ts --dry-run
npx tsx scripts/monitor-legal-documents.ts --limit=12
npx tsx scripts/monitor-legal-documents.ts --document=CPF
```

### Responsibilities

1. Read active monitored documents:
   - `Document.monitoringStatus in ["active", "unchanged", "changed", "error", "blocked"]`
   - require `officialUrl` or `canonicalUrl`.
2. For each document, independently:
   - try `HEAD`.
   - read `ETag`, `Last-Modified`, `Content-Length`.
   - if HEAD is blocked or missing enough metadata, do bounded `GET`.
   - if GET body is used, compute SHA-256 stream/buffer.
3. Detect unchanged:
   - same `etag` when present, or
   - same `lastModified` and `fileSize`, or
   - same `currentHash`.
4. Detect changed:
   - changed `etag`, or
   - changed `lastModified`/`fileSize`, or
   - changed SHA-256.
5. On unchanged:
   - update `lastCheckedAt`, `monitoringStatus = "unchanged"`, clear `lastError`.
   - dry-run prints only.
6. On changed:
   - create `DocumentVersion` with `contentHash`, `etag`, `lastModified`, `fileSize`, `sourceUrl`, `metadata`.
   - do not populate `rawText` or `originalText`.
   - create `DocumentChange` with short description and hashes.
   - create `Notification` in-app alert.
   - update `Document.currentHash`, `latestVersionHash`, `etag`, `lastModified`, `fileSize`, `lastCheckedAt`, `monitoringStatus = "changed"`, `changeSummary`.
7. On error:
   - mark only that document `monitoringStatus = "error"`, `lastError`, `lastCheckedAt`.
   - continue the batch.
8. Never fail the whole run for one source.

### Dry-run output contract

`--dry-run` must not write changes. It must print:

```text
Legal document monitor dry-run
Documents reviewed: 12
Unchanged: 10
Changed: 1
Errors: 1
Blocked: 0

Changed:
- CPF Código Penal Federal: hash changed

Errors:
- LCE Ley de Comercio Exterior: HEAD blocked, GET timed out

Blocked:
- None
```

### Implementation modules

Prefer small modules:

```text
scripts/monitor-legal-documents.ts
lib/monitoring/legalDocumentMonitor.ts
lib/monitoring/fetchMetadata.ts
lib/monitoring/hash.ts
lib/monitoring/changeDetection.ts
lib/monitoring/monitorTypes.ts
```

### Fetch constraints

- Timeout per document: 10-15 seconds.
- Use official URLs only.
- Do not follow redirects to unsafe/private URLs; reuse SSRF validation helpers where possible.
- Limit body download for hashing. If `Content-Length` is above a configured cap, rely on metadata and mark low confidence if needed.
- Do not parse PDF text.

## 10. AI Behavior

### Intent

For questions like:

- `cambios en materia penal de esta semana`
- `cambios en materia aduanal`
- `reformas recientes`
- `qué cambió en el Código Penal Federal`

The AI route must query in this order:

1. `DocumentChange` filtered by matter/document/period.
2. `DocumentVersion` if change rows are missing but versions exist.
3. `Notification` alerts if user asks about alertas.
4. Base `Document` records if no changes exist.
5. Existing search/RAG only as supplemental context.

### Required response rules

If changes exist:

```text
Encontré cambios detectados en el periodo.

1. Documento: Código Penal Federal
   Fuente: Cámara de Diputados
   Fecha de detección: ...
   Tipo: hash_changed / metadata_changed
   Resumen: ...
   Liga oficial: ...

Nivel de confianza: basado en metadatos oficiales/hash. Verificar texto oficial antes de presentar escrito.
```

If no changes exist:

```text
No encontré cambios indexados para ese periodo.
```

If documents exist but no changes:

```text
Encontré documentos base relacionados, pero no cambios detectados en el periodo consultado.
```

Never:

- inventar reformas
- inventar artículos
- inventar jurisprudencia
- decir que revisó el DOF en tiempo real si solo revisó DB local
- mandar al usuario a “usar la búsqueda” como respuesta principal

## 15. Rollback Plan

### Code rollback

```bash
git revert <commit>
npm run build
```

### Database rollback

Because migration is additive:

- Do not drop columns.
- Hide new UI routes from navigation if needed.
- Stop running monitor script.
- Leave added nullable fields unused.

### Operational rollback

If a source is unstable:

- Mark document `monitoringStatus = "blocked"` or `error`.
- Keep other documents active.
- Do not disable whole monitor.

### Data cleanup policy

No `DELETE` during rollback. If test rows must be hidden:

- mark `Document.monitoringStatus = "blocked"` or `error` only with explicit approval, or
- filter demo rows by `canonicalKey`/`shortCode`.

## 16. Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Official source blocks HEAD | Fall back to bounded GET. Mark `blocked` if both fail. |
| Large PDFs saturate memory | Use `Content-Length`, byte cap, timeout, and avoid text extraction. |
| Hash changes due to metadata-only PDF rebuild | Label as `metadata/hash change`, not substantive reform. Lawyer must verify official text. |
| False sense of legal certainty | UI and IA always include source verification warning. |
| Duplicate alert systems | Reuse `/watchlists`, `Watchlist`, `Notification`, `DocumentChange`. |
| Existing RAG flow stores text | Monitor does not call RAG/indexing. Existing ingest remains separate. |
| Render Free timeout | Manual script outside web request. |
| Migration risk | Additive-only migration, review SQL for destructive commands. |
